# Job Offer Comparator - Logic Audit and Test Coverage Plan

**Project**: Harden the `job-offer-comparator` tool so calculation changes do not silently break offer rankings, projected value, or user-visible outputs.
**Tech Stack**: TanStack Start, React 19, TypeScript, Vitest, Testing Library, localStorage persistence.
**Goal**: Audit the current calculator logic, resolve correctness-sensitive assumptions, and add a focused automated test suite that protects the core financial and decision logic of the Job Offer Comparator only.

## Current State Snapshot

- Core logic lives in `src/tools/job-offer-comparator/calculator.ts` and `src/tools/job-offer-comparator/scoring.ts`.
- Defaults and domain presets live in `src/tools/job-offer-comparator/presets.ts` and `src/tools/job-offer-comparator/constants.ts`.
- Persistence lives in `src/tools/job-offer-comparator/storage.ts`.
- UI wiring that changes calculation inputs lives in `src/tools/job-offer-comparator/page.tsx`, `src/tools/job-offer-comparator/components/offer-card.tsx`, and `src/tools/job-offer-comparator/components/advanced-offer-fields.tsx`.
- There are currently no tests for the Job Offer Comparator.

## Logic Areas That Need Explicit Protection

1. Offer projection math across 12/24/36 months.
2. Scenario adjustments for conservative, expected, and upside modes.
3. Risk-adjusted rollup and min-max finance scoring.
4. Qualitative fit scoring and blended ranking behavior.
5. One-time payouts, work costs, benefits, equity type, and cliff handling.
6. Employer preset application and user input propagation.
7. Persistence safety for saved comparator state.

## Correctness Audit Before Locking Tests

Some current behaviors should be reviewed before they are treated as the canonical spec. The implementation phase should first document and confirm these rules, then encode the confirmed behavior in tests.

- **Variable pay divide-by-zero**: `variableAnnualCash` scales with `fixedAnnualCash / offer.fixedAnnualCash`; a zero fixed salary can produce `Infinity` or `NaN`.
- **Equity cliff interpretation**: a 12-month cliff currently allows full year-1 equity because the gate is `year * 12 < equityCliffMonths`.
- **Promotion timing**: promotion is applied as a full-year multiplier starting from `ceil(months / 12)`, with no intra-year proration.
- **Tax input contract**: comparator passes `fixed + variable + employer PF` into the in-hand salary calculator; verify this matches the intended salary/tax model.
- **Negative inputs**: many numeric inputs are coerced to `0` when invalid text is entered, but negative valid numbers are still accepted and can distort outcomes.
- **Hidden-but-impactful inputs**: retention bonus, relocation support, work costs, risk deductions, expected payout, and cliff months affect math but are not all currently editable in UI.
- **Ranking consistency**: recommendation header switches between `riskAdjustedValue` and `blendedScore`; other sections still emphasize risk-adjusted winners.

## Architecture Overview

### Calculation flow

- Page state builds offers from presets, optionally loads stored state, optionally prepends baseline offer, then calls `compareOffers`.
- `compareOffers` computes expected, conservative, and upside projections per offer.
- Expected projection populates the displayed annual metrics and chart.
- Conservative and upside projections feed `riskAdjustedValue` using a 60/30/10 weighting.
- Scoring normalizes finance outcomes to 0-100, then optionally blends with qualitative inputs.
- Winners, narrative, and insights are derived from the computed outputs.

### Test strategy

- Use **pure calculator tests** for deterministic math and ranking assertions.
- Use **isolated scoring tests** for finance normalization, fit scoring, and blended score weighting.
- Use **storage tests** for malformed persisted state and fallback behavior.
- Use **light component/page tests** only where user interaction changes the compared state or applies presets.
- Use **mocked tax-calculator tests** for comparator math isolation, plus a **small integration slice** against the real in-hand calculator to ensure wiring remains valid.

## Phase 1: Spec the Calculator Surface
**Goal**: Convert the current implementation into an explicit testable specification and identify behaviors that need confirmation before they are frozen into assertions.
**Duration**: 1-2 hours

### Tasks

- Inventory every input that materially affects comparator math, including fields not currently exposed in UI.
- Write a calculation map covering annual guaranteed cash, variable pay, work cost, benefits, in-hand cash, equity, risk adjustment, finance score, fit score, and blended score.
- Mark each rule as either:
  - intended domain behavior to lock down now,
  - ambiguous behavior requiring product decision,
  - bug candidate to fix before adding final assertions.
- Define stable numeric fixtures for at least three offer profiles:
  - cash-heavy / low-risk MNC,
  - equity-heavy startup,
  - high-cost / high-fit tradeoff offer.

### Implementation details

- Prefer a reusable test fixture builder in `src/tools/job-offer-comparator/test-helpers.ts` or inline factory helpers inside test files.
- Keep fixture values intentionally simple so expected totals can be calculated by hand.
- Separate “spec confirmation” notes from the final regression assertions to avoid locking in accidental bugs.

**Verification Loop**:
- ✅ Every calculator input in `src/tools/job-offer-comparator/types.ts` is mapped to its downstream effect.
- ✅ Ambiguous or risky formulas are explicitly listed before assertions are written.
- ✅ A minimal reusable fixture set exists on paper for all later tests.

## Phase 2: Add Core Projection Tests
**Goal**: Protect the deterministic 1/2/3-year value calculations for a single offer under controlled assumptions.
**Duration**: 2-3 hours

### Tasks

- Add `calculator.test.ts` for single-offer projection cases through `compareOffers` using two-offer comparisons where required.
- Mock `#/tools/inhand-salary/calculator` in most tests so the comparator’s own formulas can be asserted cleanly.
- Cover these high-value cases:
  - benefits contribute only when enabled,
  - disabled benefits do not count,
  - work costs reduce realized value,
  - one-time payouts only affect year 1,
  - notice/clawback deductions only affect year 1,
  - employee/employer PF impact the annual values as currently intended,
  - partial first-year increment via `nextIncrementMonth`,
  - promotion uplift begins in the correct modeled year,
  - steady-state annual value comes from year 3,
  - 24-month and 36-month totals equal the sum of projection points,
  - realized yearly value never drops below zero.

### Implementation details

- Use a tax stub that returns a predictable yearly in-hand value so expected totals are algebraic and readable.
- Add one or two tests with the real in-hand calculator left unmocked to verify comparator-to-tax-calculator integration still produces finite numbers.
- Prefer `toBeCloseTo` for currency totals that involve compounding.

**Verification Loop**:
- ✅ Projection tests cover the main money-moving inputs that users can change.
- ✅ Aggregate metrics (`firstYearRealizedValue`, `value24Months`, `value36Months`, `steadyStateAnnualValue`) are derived correctly.
- ✅ No projection test relies on opaque manual inspection of raw objects.

## Phase 3: Add Scenario, Risk, and Ranking Tests
**Goal**: Protect the multi-scenario decision logic that determines winners and rankings.
**Duration**: 2-3 hours

### Tasks

- Add assertions proving conservative, expected, and upside modes apply the configured multipliers from `SCENARIO_CONFIG`.
- Verify risk-adjusted value uses exact `expected * 0.6 + conservative * 0.3 + upside * 0.1` weighting.
- Verify chart rows reflect the selected expected scenario projection only.
- Test winner selection for:
  - best cash now,
  - best 36 months,
  - best risk-adjusted,
  - best overall when fit inputs are enabled.
- Add tests for equal-risk-adjusted offers so finance normalization returns `100` for all offers.

### Implementation details

- Create two-offer and three-offer fixtures where the leading offer changes by metric.
- Assert both raw computed values and the declared `winners` object so ranking regressions are caught early.
- Include at least one test where `config.scenario` is not `expected` to ensure the active lens truly drives displayed metrics.

**Verification Loop**:
- ✅ Scenario multipliers are locked to the current intended model.
- ✅ Risk-adjusted weighting cannot drift unnoticed.
- ✅ Winner selection matches the computed metric values in every tested lens.

## Phase 4: Add Fit Scoring and Preset Tests
**Goal**: Protect qualitative scoring, blended ranking, and employer preset application.
**Duration**: 2 hours

### Tasks

- Add `scoring.test.ts` for:
  - qualitative average to 0-100 mapping,
  - lower and upper bound handling,
  - finance-only mode returning `financeScore` as `blendedScore`,
  - normalized weight handling when finance and fit weights sum to 100 or other values.
- Add tests around `applyEmployerPreset` to verify employer-type changes update only the intended fields.
- Add a light UI or interaction test for `OfferCard` or page-level handlers verifying employer-type selection updates preset fields in the offer state.
- Add a test covering baseline inclusion so enabling current role baseline changes the compared set.

### Implementation details

- Keep UI tests narrow: assert state-driven behavior, not styling or collapsible animation.
- Prefer one page-level behavior test over many component implementation-detail tests.
- Preserve separation between pure-domain tests and UI wiring tests.

**Verification Loop**:
- ✅ Fit scoring and blended ranking behave predictably across weight changes.
- ✅ Employer preset changes cannot silently drift from their intended defaults.
- ✅ Baseline and employer-type interactions are protected by at least one user-facing test.

## Phase 5: Add Edge-Case and Persistence Tests
**Goal**: Protect the tool from malformed inputs and corrupted persisted state.
**Duration**: 2-3 hours

### Tasks

- Add regression tests for problematic numeric edge cases:
  - zero fixed cash with non-zero variable target,
  - zero or negative totals producing no `NaN`/`Infinity`,
  - extreme benefit or cost values still returning finite outputs,
  - cliff and equity-type combinations (`none`, `rsu`, `esop`).
- Add storage tests for:
  - invalid JSON,
  - missing required sections,
  - partial baseline object,
  - missing open-state maps falling back to `{}`.
- Add input coercion tests for fields using `toNumber`/`toInt` where invalid text should resolve to `0`.

### Implementation details

- If implementation chooses to clamp or sanitize values in calculator layer, update the spec and tests together.
- Favor tests that assert “finite safe result” for unstable edge cases and exact totals for stable business rules.
- Keep storage tests isolated from React rendering where possible.

**Verification Loop**:
- ✅ Comparator handles malformed state and edge inputs without crashing.
- ✅ The known divide-by-zero path is either fixed and asserted, or explicitly guarded by a failing test before the fix.
- ✅ Persistence tests guarantee safe fallback behavior for old or bad browser state.

## Phase 6: Final Verification and Coverage Review
**Goal**: Ensure the suite is maintainable, fast, and aligned with the intended product model.

### Tasks

## Recommended Test Matrix

### Calculator regression tests

- Expected scenario baseline computation with simple fixture.
- Partial increment effect based on `nextIncrementMonth`.
- Promotion timing and uplift effect at 24-36 months.
- Work-cost subtraction and benefit inclusion/exclusion.
- One-time joining/retention/relocation impact only in year 1.
- Risk deductions only in year 1.
- RSU vs ESOP vs none equity behavior.
- Equity cliff behavior at 0, 12, and more-than-12 months.
- Realized values clamped at zero.
- Finite outputs for zero fixed salary and extreme/invalid numeric combinations.

### Ranking/scenario tests

- Conservative scenario lowers variable/equity/growth versus expected.
- Upside scenario raises them and advances promotion timing.
- Risk-adjusted weighting is 60/30/10.
- Winner objects follow computed metrics.
- Chart rows use active selected scenario outputs.

### Scoring tests

- Finance normalization with clear min/max spread.
- Finance normalization when all offers tie.
- Qualitative score mapping from six inputs.
- Fit-enabled blended ranking with custom weights.
- Finance-only mode ignores fit for recommendation.

### Storage/UI wiring tests

- Malformed localStorage returns `null` safely.
- Missing optional maps default to empty objects.
- Employer-type change applies preset fields.
- Baseline toggle includes current role in comparison set.
- Invalid numeric input text resolves to `0` in controlled handlers.
