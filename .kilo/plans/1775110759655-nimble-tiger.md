# Job Offer Comparator - Development Plan

**Project**: Build a privacy-first, mobile-friendly Job Offer Comparator for PayDisha that helps salaried employees compare 2-3 offers without being overwhelmed.
**Tech Stack**: TanStack Start, TanStack Router, React, Tailwind CSS v4, shadcn/ui, existing chart/tooltip/collapsible/table components, local-only browser storage.
**Goal**: Ship an MVP that turns messy offer letters into a clear decision: best cash now, best long-term upside, best lifestyle fit, and the trade-offs behind each.

## Product Intent

This tool should feel native to PayDisha's existing philosophy:
- Fast, practical, and decision-oriented rather than content-heavy.
- Private by default: no sign-up, no tracking, no server-side storage.
- Progressive disclosure: quick compare first, advanced realism only when needed.
- Mobile-first: most users will use this tool on their phone.
- Honest outputs: explain assumptions and trade-offs instead of hiding them behind one opaque score.

### Key market takeaway

The whitespace is not "another salary calculator." The opportunity is an **India-native offer decision engine** that combines:
- CTC normalization
- tax-aware realized cash
- joining/bonus/equity timing
- city and work-mode costs
- benefits monetization
- qualitative fit inputs

## Recommended MVP Positioning

**Product promise**: "Compare 2-3 job offers the way your real life experiences them - not the way HR writes the CTC."

**Primary user**: A salaried employee with 2-3 job offers, typically comparing present job vs new offers or multiple competing offers.

**Core questions the tool must answer**:
- Which offer gives me the best monthly life now?
- Which offer is strongest over the next 24-36 months?
- Which offer looks good on paper but is risky in reality?
- How much of the difference is caused by taxes, city costs, variable pay, or equity assumptions?
- If I care about culture/growth/WLB too, does the winner change?

## MVP Scope

### In scope
- Compare **2 offers by default**, with optional third offer.
- Optional **current job baseline** for effective-hike context.
- Finance lens: fixed pay, variable pay, joining bonus, employer retirement contribution, equity/ESOP/RSU value, relocation support, recurring personal work costs.
- Timing lens: first-year vs steady-state vs 24/36-month realized value.
- Qualitative lens: role fit, growth, brand/resume value, work-life sustainability, manager/culture confidence.
- Scenario lens: conservative / expected / upside.
- Mobile-friendly, browser-only, local-only persistence.

### Explicitly out of scope for MVP
- Offer-letter document upload/parser.
- Live market salary benchmarking API.
- Company reviews or scraped culture data.
- Automated negotiation scripts.
- Support for hourly/contract/freelance offers.

## UX Strategy

### Low-barrier input philosophy

The tool should not open with 20 fields per offer. Use a **two-layer input model**:

1. **Quick Compare (required)**
   - Offer label / company name
   - Work city
   - Work mode: remote / hybrid / on-site
   - Annual fixed cash
   - Variable pay target
   - Joining bonus
   - Equity value (optional in quick mode)

2. **Add Realism (optional advanced)**
   - Tax regime
   - Monthly employee PF
   - Appraisal month / next increment cycle
   - Expected annual increment
   - Promotion expectation window
   - Bonus payout confidence
   - Equity type: RSU / ESOP / none
   - Vesting / cliff notes
   - Relocation support or rent delta
   - Commute / travel / meal / remote setup costs
   - Benefits toggles with optional value overrides
   - Clawback / notice buyout / retention lock-in

### Smart defaults to reduce input burden

- Default to **2 offers** and reveal "Add third offer" as an action.
- Pre-fill common assumptions by **company archetype**: Startup / Late-stage / MNC / Global Tech.
- Benefits should start as **chips/toggles**, not raw numeric fields.
- Use **reasonable default ranges** for increment cycle, variable payout certainty, and benefit values, with override capability.
- Keep qualitative factors separate from money math; do not pretend culture can be converted to rupees.

### Output philosophy

Avoid a single opaque winner. Show:
- **Best for monthly life now**
- **Best over 24-36 months**
- **Best risk-adjusted offer**
- **Best overall fit** (only if the user uses qualitative inputs)

## Calculation Strategy

### Compensation normalization

Break every offer into comparable buckets:
- Guaranteed recurring cash
- At-risk recurring cash
- One-time cash
- Employer-paid retirement value
- Equity value
- Personalized recurring work costs
- Personalized monetized benefits

### Core formulas

- **Guaranteed annual cash** = fixed annual cash + guaranteed recurring allowances
- **Expected annual cash** = guaranteed annual cash + (variable target x payout confidence)
- **First-year realized value** = expected annual cash + joining/relocation/one-time cash + year-1 benefit value - annual work costs
- **Steady-state annual value** = expected annual cash + annualized equity + recurring benefit value - recurring work costs
- **24/36-month value** = year-by-year expected value with increment assumptions and timing events
- **Risk-adjusted value** = conservative scenario weighting on bonus/equity/promotion assumptions

### Important modeling rules

- Keep **monthly in-hand** focused on recurring cash, not inflated by one-time bonuses.
- Show **guaranteed vs at-risk** compensation separately.
- Public RSUs and private ESOPs must not be treated the same.
- Qualitative score must remain visibly separate from finance score.
- If assumptions are missing, favor transparent defaults over fake precision.

## Benefits Engine

### MVP benefits categories

Use toggle-first inputs with optional value override:
- Health insurance (self)
- Health insurance (parents/family)
- Meal card / food allowance
- Phone / internet reimbursement
- Learning budget
- Wellness budget
- Transport / cab support
- Remote setup support
- Childcare / family support
- Extra leave / flexible work policy

### Benefits UX rule

The UI should first ask **"Which of these do you actually value?"** then allow edits. This makes benefits personal rather than decorative.

## Qualitative Decision Layer

### Why it matters

Candidates rarely choose solely on cash. The tool should let users layer in optional judgment without diluting trust.

### Suggested qualitative inputs

- Role excitement
- Manager confidence
- Work-life sustainability
- Growth/promotion confidence
- Brand/resume value
- Job security confidence

### Scoring rule

- Finance score always available.
- Fit score appears only after qualitative inputs are used.
- Blended score is optional and user-weighted.

## Mobile-First UX Rules

- Keep the page stacked by default; desktop can use the existing left-input / right-results pattern.
- Use short cards and tabs before tables.
- Ensure comparison tables scroll horizontally without breaking the layout.
- Show the verdict and winner cards before any chart.
- Use collapsible "Advanced inputs" sections for each offer.
- Make offer cards easy to duplicate, rename, collapse, and delete.

## Phased Implementation

## Phase 1: Product Framing and Calculation Contracts
**Goal**: Lock the MVP scope, offer schema, and comparison lenses before UI work.
**Duration**: 2-3 hours

### Tasks
1. Define the offer schema and all derived metrics.
2. Split inputs into required vs advanced.
3. Finalize default assumptions for company archetypes and benefit valuation.
4. Decide conservative / expected / upside scenario multipliers.

### Implementation Details
- Create typed interfaces for offers, presets, benefits, qualitative inputs, and comparison results.
- Document which fields are user-entered, defaulted, or derived.
- Keep current-job benchmarking optional.

**Verification Loop**:
- [ ] Every core offer component maps to one normalized bucket.
- [ ] Required fields are few enough for a sub-5-minute first comparison.
- [ ] Scenario assumptions are explicit and human-readable.
- [ ] Finance score and fit score are intentionally separate.

## Phase 2: Build the Comparison Engine
**Goal**: Create a reliable calculation layer for offer normalization and multi-horizon comparison.
**Duration**: 4-6 hours

### Tasks
1. Implement compensation normalization and recurring/one-time separation.
2. Implement year-1, 24-month, and 36-month projections.
3. Add scenario modeling for bonus/equity/promotion uncertainty.
4. Build insight generation from major deltas.

### Implementation Details
- Reuse existing salary/tax concepts where practical, but do not block MVP on a full shared-finance refactor.
- Build deterministic helpers for monthly take-home, benefit value, work costs, and risk-adjusted value.
- Generate structured insights like "Offer B wins because the city-cost gap is smaller than the salary gap."

**Verification Loop**:
- [ ] Two offers with identical assumptions produce identical results.
- [ ] One-time bonuses do not artificially inflate monthly in-hand.
- [ ] Conservative and upside scenarios visibly change risk-sensitive offers.
- [ ] Year-1 vs steady-state differences are explained clearly.

## Phase 3: Quick Compare Input Experience
**Goal**: Make the tool feel easy and fast on the first screen.
**Duration**: 4-5 hours

### Tasks
1. Build the default 2-offer quick compare flow.
2. Add optional third offer and optional current-job baseline.
3. Add archetype presets and advanced sections.
4. Add local-only persistence and reset actions.

### Implementation Details
- Desktop: sticky left panel; mobile: stacked cards.
- Each offer card should support rename, duplicate, collapse, and delete.
- Include a strong privacy note near the inputs.
- Persist locally only after hydration, mirroring existing salary-tool behavior.

**Verification Loop**:
- [ ] A first-time user can compare 2 offers without opening advanced fields.
- [ ] Mobile users can complete all required fields without horizontal layout issues.
- [ ] Offer cards are manageable when 3 offers are present.
- [ ] Reset and local-storage persistence behave predictably.

## Phase 4: Results, Verdicts, and Scenario UX
**Goal**: Turn raw numbers into a trustworthy decision surface.
**Duration**: 5-6 hours

### Tasks
1. Build the verdict header and ranking cards.
2. Add scenario tabs and comparison charts.
3. Add the detailed component comparison table.
4. Add narrative insight blocks and timing warnings.

### Implementation Details
- Use cards first, charts second, tables last.
- Emphasize winner by lens rather than one global winner only.
- Prefer compact bar/stacked charts over dense dashboards.
- Timing insights should call out appraisal-cycle misses, bonus timing, and equity cliff risks.

**Verification Loop**:
- [ ] Users can tell the best offer for cash now within 5 seconds.
- [ ] Users can tell the best offer for long-term upside without opening the table.
- [ ] Scenario tabs change both numbers and explanation copy.
- [ ] Detailed tables remain usable on mobile through horizontal scroll.

## Phase 5: Qualitative Fit Layer and Weighted Decisioning
**Goal**: Add optional non-financial decision support without compromising trust.
**Duration**: 3-4 hours

### Tasks
1. Add qualitative inputs and optional weighting.
2. Generate fit score and blended score.
3. Show how the winner changes when fit is considered.
4. Add explanatory copy around subjectivity.

### Implementation Details
- Keep fit inputs collapsed by default.
- Make it explicit that this is the user's judgment layer, not objective truth.
- Highlight if finance winner and fit winner diverge.

**Verification Loop**:
- [ ] The tool still works fully without qualitative inputs.
- [ ] Fit score does not overwrite or hide finance score.
- [ ] Blended score is understandable and adjustable.
- [ ] Divergence between money and fit creates a useful insight, not confusion.

## Phase 6: Polish, Accessibility, and Release Readiness
**Goal**: Align the tool with PayDisha quality standards and existing design language.
**Duration**: 3-4 hours

### Tasks
1. Polish mobile spacing, empty states, and loading states.
2. Validate copy, accessibility, and local-trust messaging.
3. Add homepage tool-card metadata and route wiring.
4. Run checks and verify no regressions.

### Implementation Details
- Use semantic labels, tooltips, and helper text for ambiguous terms like ESOP or payout confidence.
- Add empty results state that explains what users get before they fill data.
- Keep all state local to browser storage only.

**Verification Loop**:
- [ ] The tool is usable on a 375px mobile viewport.
- [ ] Keyboard navigation and screen-reader labels are present for interactive controls.
- [ ] Trust messaging clearly states local-only handling of sensitive offer data.
- [ ] `pnpm check` passes without formatting or type errors.

## Technical Notes

- Reuse existing route pattern used by `buy-vs-rent` and `salary-growth` by keeping the route file thin and the full page under `src/tools/...`.
- Reuse existing visual primitives and utility classes from `src/styles.css`.
- Prefer pure functions for all financial calculations so scenario testing remains easy.
- Store assumptions and presets in dedicated constants/presets files to avoid mixing logic and UI.
- Keep string explanations generated from structured result flags rather than ad-hoc inline conditions.

## Recommended MVP Defaults

- Offer count: 2 by default, third optional.
- Baseline: current job optional.
- Scenario tabs: Conservative / Expected / Upside.
- Company archetypes: Startup / Late-stage / MNC / Global Tech.
- Qualitative weighting: off by default.
- Persistence: local storage with explicit reset and privacy note.

## Success Criteria

- User can compare 2 offers end-to-end in under 5 minutes on mobile.
- Tool explains the difference between CTC and realized value clearly.
- Results answer both short-term and medium-term decision questions.
- Variable pay, equity, and location costs are visible instead of hidden inside CTC.
- Finance and subjective-fit outputs remain separate and trustworthy.
- The page feels visually consistent with existing PayDisha tools.
